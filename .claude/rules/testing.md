# テスト規約

コードの変更を行う場合、**対応するユニットテストを必ず作成・更新すること**。すべてのコード変更にはテストを含めること。

- **新機能追加時:** 新しいクラス・メソッドに対するテストを作成する
- **バグ修正時:** 修正内容を検証するリグレッションテストを作成する
- **リファクタリング時:** 既存テストが通ることを確認し、必要に応じてテストも更新する
- **テスト配置:** テストディレクトリ（`src/test/`, `tests/`, `__tests__/` 等）にソースと対応する構成で配置する
- **テスト命名:** テスト対象が明確に分かる命名にする（例: `<対象クラス名>Test`, `<対象モジュール名>.test`）
- **tasklist.md への記載:** ステアリングワークフローの tasklist.md には、実装タスクとセットでテスト作成タスクを必ず含めること

**例外:** UI コンポーネント（GUI ベースの設定画面等）や、外部サービスとの結合が必須で単体テストが困難なクラスは、テスト作成を省略してよい。ただし、その場合は tasklist.md にテスト省略の理由を明記すること。

## 検証可能ゴール変換（Goal-Driven Execution）

> 出典: [Karpathy Guidelines](https://zenn.dev/kotai/articles/050f0220c3f4de) 原則 4。

ユーザーから曖昧な指示（「動くようにして」「バリデーション追加」等）を受けたとき、**実装に着手する前にテストで表現可能な完了条件に変換**すること。これにより Claude が自律的に検証ループを回せるようになる。

| 曖昧な指示 | 検証可能ゴール |
|----------|---------------|
| 「バリデーション追加」 | 不正入力で失敗するテストを作成 → 実装後に通る |
| 「バグ修正」 | バグを再現するリグレッションテストを作成 → 修正後に通る |
| 「リファクタリング」 | 変更前にテストが全件パスすることを確認 → 変更後も同一テストがパス |
| 「パフォーマンス改善」 | ベンチマーク値を測定 → 目標値を上回ることをテストで保証 |
| 「機能 X を追加」 | X の受け入れ条件をテストとして書き、Red → Green を確認 |

**強い基準と弱い基準:**
- **強い基準**（テストで表現可能）: Claude が自律的に「実装 → テスト実行 → 失敗なら修正」のループを回せる
- **弱い基準**（「動くようにしろ」等）: 常時人間の確認が必要。Plan Mode / steering の計画段階で強い基準に変換すること

**複数ステップタスクの場合:** 各ステップに検証方法を併記する（例: `1. パーサ実装 → 検証: parser.test.ts 全件パス`）。

## 自己検証

実装完了後、コミット前に以下の検証を必ず実行すること:

1. **テスト実行**: 変更に関連するテストをすべて実行し、全件パスすることを確認する
2. **型チェック**: プロジェクトに型チェックツール（TypeScript, mypy 等）がある場合、型エラーがないことを確認する
3. **リンター**: プロジェクトにリンター（ESLint, Ruff 等）がある場合、警告・エラーがないことを確認する
4. **ビルド**: プロジェクトにビルドコマンドがある場合、ビルドが成功することを確認する

**失敗時の対応:**
- 検証で失敗が発生した場合、自律的に原因を特定し修正すること
- 修正後、再度すべての検証を実行すること
- 3回修正しても解決しない場合、ユーザーに報告してアプローチを相談すること
