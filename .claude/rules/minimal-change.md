# 最小変更原則（YAGNI + Surgical Changes）

> 出典: [Karpathy Guidelines](https://zenn.dev/kotai/articles/050f0220c3f4de) の原則 2（Simplicity First）・原則 3（Surgical Changes）を本テンプレート用に再構成したもの。

ユーザー要求を満たす **最小限のコード** を、**最小限の差分** で提供すること。レビュー負荷を下げ、本質的な変更を明確にするための規律。

## YAGNI（投機的実装の禁止）

要求された以上の機能を作らない。

- **禁止:** 要求されていないエラーハンドリング、起こり得ないシナリオへの対応、将来必要になるかもしれない設定可能性・拡張ポイント
- **禁止:** 一度しか使われないコードへの抽象化（ストラテジーパターン / プラグイン機構等）
- **禁止:** 「良い設計パターン」の無批判な適用（訓練データに頻出するからという理由だけで導入しない）
- **推奨:** シンプルな実装で要求が満たせるなら、それで止める（`fs.readFileSync` + `JSON.parse` で十分なら、それで済ます）
- **自問:** シニアエンジニアが「やりすぎ」と言わないか？ 200 行が 50 行で済むなら書き直す

**バウンダリでの例外:** ユーザー入力 / 外部 API のような信頼境界では、入力検証を必ず行う。社内コード間の呼び出しは型と契約を信頼してよい。

## Surgical Changes（隣接コードに触れない）

変更は**ユーザー要求に直接紐づく行**のみに限定する。

### 既存コード編集時

- **禁止:** 隣接コードのリファクタリング、コメント・フォーマットの「ついで改善」
- **禁止:** 動作している箇所のスタイル変更
- **必須:** 既存のコードスタイルに合わせる（個人的な好みより優先）
- **禁止:** 関係のないデッドコードの削除（発見した場合は言及のみ、削除はしない）

### 自分の変更で生じたコードの処理

- **必須:** 自分の変更で不要になった import / 変数 / 関数は削除する
- **禁止:** 元から存在していたデッドコードは削除しない（明示要求がない限り）

### 検証基準

コミット前に以下を自問すること:

- 変更した全行が、ユーザーリクエストに直接紐づいているか？
- 「ついでに直した」行が混入していないか？

混入している場合は、別コミット / 別 PR に分離する。
