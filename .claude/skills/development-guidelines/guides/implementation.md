# 実装ガイドライン（コーディング規約）

プロジェクト固有の規約が `docs/development-guidelines.md` にある場合はそちらを優先する。
このファイルは汎用的な参考資料として使用する。

## 1. 全般

### 命名規則

プロジェクトの言語・フレームワークに応じた命名規則を採用する。
CLAUDE.md に規定がある場合はそちらに従うこと。

**一般的な指針:**

| 対象 | 規則 | 例 |
|------|------|-----|
| クラス / 型 | 言語の慣習に従う | `UserService`, `Transaction` |
| 関数 / メソッド | 言語の慣習に従う | `calculateTotal`, `parseInput` |
| 変数 | 言語の慣習に従う | `userName`, `itemCount` |
| 定数 | UPPER_SNAKE_CASE が一般的 | `MAX_RETRY_COUNT` |
| ファイル名 | プロジェクトの慣習に従う | - |

### ファイル構成

- 1ファイル1責務を徹底する
- ファイルの分割粒度はプロジェクトの慣習に従う

### エラーハンドリング

- 言語が提供する型安全なエラーハンドリング機構を活用する
- 例外は境界（IO、外部API呼び出し）でのみ catch する
- 内部ロジックは型で安全性を担保する

## 2. コード品質

### 読みやすさ

- 意図が明確に伝わるコードを書く
- 複雑な処理にはコメントで「なぜそうしているか」を説明する
- ネストが深くなりすぎないよう、早期リターンやガード節を活用する

### 単一責任の原則

- 関数は1つのことだけを行う
- クラス / モジュールは1つの責務だけを持つ
- 重複コードは適切に共通化する

### 不変性

- 可能な限り不変（immutable）なデータ構造を使用する
- 副作用はシステムの境界に押し出す

## 3. テスト規約

### テストの構造

- Arrange-Act-Assert パターンを使用する
- テスト名は対象の振る舞いを記述する（`should ...` または `〜すべき`）

### テストカバレッジ

- ビジネスロジックは必ずテストを書く
- エッジケース（空リスト、null/None、0件、境界値）をテストする
- エラーケースのテストも含める

## 4. コメント規約

CLAUDE.md のコメント規約に従う。規約がない場合:

- **ファイル先頭**: このファイルの役割・責務の概要を記載
- **複雑な処理の直前**: 何をしているか、なぜそうしているかを記載
- 自明なコードにはコメントを付けない
