# 開発プロセスガイドライン

プロジェクト固有のプロセスが `docs/development-guidelines.md` にある場合はそちらを優先する。
このファイルは汎用的な参考資料として使用する。

## 1. 開発フロー概要

```
要件定義 → 影響分析 → 設計 → 実装 → テスト → レビュー → マージ
    │           │        │       │        │        │
    ▼           ▼        ▼       ▼        ▼        ▼
 .steering/  docs/確認  design.md コード   品質チェック docs/更新
 requirements.md
```

## 2. 作業開始前の準備

### 2.1 影響分析

新機能の追加や既存機能の修正を行う前に、以下を確認する:

1. **関連ドキュメントの確認**
   - `docs/architecture.md` でアーキテクチャ構成を確認
   - `docs/functional-design.md` で既存の画面仕様・API仕様を確認
   - `docs/glossary.md` で用語の定義を確認

2. **影響範囲の特定**
   - 変更対象のモジュール
   - 依存しているモジュール（上流・下流）
   - 影響を受けるテスト

### 2.2 作業ディレクトリの作成

```

├── requirements.md    # 要件定義

```

## 3. 実装プロセス

### 3.1 実装の順序

プロジェクトのアーキテクチャに応じた順序で実装する。一般的な指針:

1. **型定義・モデル（Domain層）**
2. **ビジネスロジック（Core層）**
3. **データアクセス（Infrastructure層）**
4. **API / プレゼンテーション（Presentation層）**
5. **テスト**

### 3.2 コミット粒度

- 1つの論理的な変更 = 1コミット
- CLAUDE.md にコミット規約がある場合はそれに従う
- コミットメッセージは変更の「なぜ」を伝える

### 3.3 品質チェック

各実装ステップ完了後に、プロジェクトのビルドコマンドとテストコマンドを実行する。

**品質基準:**
- ビルドがエラーなしで完了
- テストが全件パス
- 新規コードにコメント規約に沿ったコメントが付与されている

## 4. ドキュメント管理

### 4.1 Single Source of Truth

- 同じ情報を複数箇所に書かない
- 参照リンクで繋ぐ
- CLAUDE.md に詳細が記載されている場合は、そちらを参照する

### 4.2 docs/ の更新タイミング

| イベント | 更新対象 |
|---------|---------|
| 新機能追加 | `functional-design.md`, `architecture.md` |
| API変更 | `functional-design.md` |
| アーキテクチャ変更 | `architecture.md` |
| 新しい用語の追加 | `glossary.md` |
| ディレクトリ構成変更 | `repository-structure.md` |
| 開発プロセス変更 | `development-guidelines.md` |

### 4.3 .steering/ の管理

- 作業開始時に作成
- 作業完了後もアーカイブとして残す（削除しない）
- 進捗は `tasklist.md` に記録する

## 5. レビュー観点

### コードレビューチェックリスト

- [ ] プロジェクトのコーディング規約に準拠しているか
- [ ] 命名規則が統一されているか
- [ ] エラーハンドリングが適切か
- [ ] テストが追加されているか
- [ ] コメント規約に従っているか
- [ ] ドキュメントが更新されているか
- [ ] セキュリティ上の問題がないか

## 6. トラブルシューティング

### ビルドエラー

1. エラーメッセージを確認する
2. 依存関係が正しくインストールされているか確認する
3. ビルドキャッシュをクリアして再ビルドする

### テスト失敗

1. エラーメッセージを確認
2. 関連するソースコードを読む
3. 個別テストを実行して原因を特定する
4. デバッグ出力を追加して原因特定
