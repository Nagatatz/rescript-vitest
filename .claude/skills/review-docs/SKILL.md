# ドキュメントレビュー

プロジェクトドキュメントの品質を5つの評価基準でレビューし、構造化されたレポートを出力する。

## 使い方

- `/review-docs` — `docs/` 配下の全 `.md` ファイルをレビュー
- `/review-docs docs/architecture.md` — 指定ファイルのみレビュー
- `/review-docs docs/architecture.md docs/functional-design.md` — 複数ファイル指定も可

## レビュー手順

### ステップ1: 対象ドキュメントの特定

引数が指定されている場合はそのファイルを対象とする。引数がない場合は Glob で `docs/**/*.md` を検索し、見つかった全ファイルを対象とする。

### ステップ2: プロジェクトコンテキストの把握

CLAUDE.md を読み、以下を確認する:

- プロジェクトの概要と目的
- アーキテクチャの構成
- コーディング規約
- ドキュメント管理の原則

### ステップ3: 各ドキュメントを5つの評価基準で評価

対象ドキュメントをすべて Read で読み込み、以下の5基準で評価する。各基準は1〜5のスコアで採点する。

### ステップ4: レビューレポートの生成

後述のレポートフォーマットに従い、レビュー結果を出力する。

### ステップ5: 改善の優先順位を提示

指摘事項を影響度順に並べ、どこから着手すべきかを明示する。

## 5つの評価基準

### 1. 完全性（Completeness）

ドキュメントに必要な情報が漏れなく記載されているか。

**チェック項目:**
- 必須セクション（目的、スコープ、前提条件、制約）が存在するか
- 記載されている情報は十分に詳細か
- 「TBD」「TODO」「後で書く」などの未完成箇所がないか
- 前提条件や依存関係が明記されているか
- 対象読者が想定され、その読者に必要な情報が揃っているか

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | 全セクション完備。情報の抜け漏れなし |
| 4 | ほぼ完備。軽微な情報の不足のみ |
| 3 | 主要セクションはあるが、一部に情報不足 |
| 2 | 重要な情報が複数欠落 |
| 1 | 骨格のみ。大半の情報が未記載 |

### 2. 明確性（Clarity）

読者が迷わず理解できる記述になっているか。

**チェック項目:**
- 用語が一貫して使用されているか（用語集 `docs/glossary.md` があれば照合）
- 専門用語・略語は初出時に定義されているか
- 図表やコード例で具体的に説明されているか
- 文章構造が論理的で読みやすいか
- 曖昧な表現（「適宜」「必要に応じて」「など」の多用）がないか

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | 初学者でも迷わず理解可能。具体例・図表が豊富 |
| 4 | 概ね明確。一部に曖昧な表現あり |
| 3 | 理解可能だが、曖昧な箇所が散見される |
| 2 | 専門知識がないと理解が困難な箇所が多い |
| 1 | 全体的に曖昧で、読者の解釈に依存する |

### 3. 一貫性（Consistency）

ドキュメント内・ドキュメント間・実装コードとの整合が取れているか。

**チェック項目:**
- ドキュメント内で同じ概念に異なる名称を使っていないか
- 他ドキュメントとの記述が矛盾していないか（Grep で関連ドキュメントを横断検索）
- 実装コードとの整合性（Grep でコードベースを検索し、型名・API パス・ディレクトリ構造等を照合）
- 命名規則・フォーマットが統一されているか
- 日付・バージョン情報が最新か

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | 完全に一貫。コードとも整合 |
| 4 | ほぼ一貫。軽微な不整合のみ |
| 3 | 概ね一貫だが、一部に矛盾あり |
| 2 | 複数の矛盾が存在。コードとの乖離あり |
| 1 | 一貫性がなく、信頼性に欠ける |

### 4. 実装可能性（Implementability）

ドキュメントの内容がそのまま実装に使える具体性を持っているか。

**チェック項目:**
- API仕様はリクエスト/レスポンスの型まで定義されているか
- 画面仕様は具体的な挙動が記述されているか
- 技術的な記述は正確か（存在しないライブラリや非推奨APIを参照していないか）
- 依存関係やセットアップ手順が明記されているか
- 手順に従えば再現可能か

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | そのまま実装着手可能。型定義・手順が具体的 |
| 4 | 概ね実装可能。一部に確認が必要な箇所あり |
| 3 | 方針は分かるが、詳細の確認が必要 |
| 2 | 抽象的で、大幅な追加調査が必要 |
| 1 | 実装の指針にならない |

### 5. 測定可能性（Measurability）

成功基準やテスト条件が明確に定義されているか。

**チェック項目:**
- 機能の成功基準・完了条件が定義されているか
- テスト可能な形で要件が記述されているか
- 受入条件（Acceptance Criteria）が明記されているか
- 数値目標やKPIがあれば、測定方法が記載されているか
- 「正しく動作する」のような曖昧な基準になっていないか

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | 全要件にテスト可能な成功基準あり |
| 4 | 主要要件に基準あり。一部に曖昧さ |
| 3 | 一部の要件のみ基準あり |
| 2 | 基準がほぼなく、判定が主観的 |
| 1 | 成功基準が一切定義されていない |

## レポートフォーマット

以下の形式でレビュー結果を出力する。

### 評価サマリー

全ドキュメントのスコアを一覧表にする。

```
| ドキュメント | 完全性 | 明確性 | 一貫性 | 実装可能性 | 測定可能性 | 総合 |
|-------------|--------|--------|--------|-----------|-----------|------|
| architecture.md | 4 | 5 | 3 | 4 | 3 | 3.8 |
| functional-design.md | 3 | 4 | 4 | 3 | 2 | 3.2 |
```

総合スコアは5基準の平均値（小数点第1位まで）。

### 良い点

各ドキュメントの優れている点をリストアップする。具体的にどの記述が良いかを示す。

```
- **architecture.md**: レイヤー構成が図表で明確に示されており、各モジュールの責務が具体的
- **functional-design.md**: API仕様がリクエスト/レスポンス例付きで実装しやすい
```

### 指摘事項

改善が必要な箇所を [必須] / [推奨] に分類して列挙する。

```
#### [必須] 指摘事項のタイトル

- **対象**: ドキュメント名 > セクション名
- **基準**: 該当する評価基準名
- **理由**: なぜ問題なのかの説明
- **改善案**: 具体的な改善方法
- **改善前**: 現在の記述（引用）
- **改善後**: 改善後のイメージ（提案）
```

[必須] と [推奨] の分類基準:
- **[必須]**: 実装に支障がある、または誤解を招く重大な問題
- **[推奨]**: 品質向上のための改善提案。なくても実装は可能

### 改善の優先順位

指摘事項を影響度の高い順に並べ、着手順序を提示する。

```
1. **最優先**: [必須] の指摘事項（実装ブロッカー）
2. **高**: [必須] の指摘事項（誤解リスク）
3. **中**: [推奨] の指摘事項（品質向上）
4. **低**: [推奨] の指摘事項（体裁改善）
```
