# コンテキスト管理ガイドライン

## 戦略的コンパクトのタイミング

以下のタイミングで `/compact` を実行し、コンテキストを整理する:

- **探索フェーズ終了後**: コードベースの調査・分析が完了し、実装に移る前
- **マイルストーン完了後**: 機能の実装やバグ修正が一段落した後
- **エラー解決後**: デバッグに多くのコンテキストを消費した後
- **方針転換時**: 異なるアプローチに切り替える際、不要になった試行の情報を整理

`/compact` には指示を付けて、保持すべき情報を明示できる:
- `/compact API変更の内容を保持` — 特定の作業コンテキストを優先的に残す
- `/compact 変更したファイル一覧とテストコマンドを保持` — 実装状態を残す

部分的なコンパクションには `/rewind`（`Esc+Esc`）から「Summarize from here」を選択する。選択したメッセージ以降のみを要約し、それ以前のコンテキストはそのまま残る。

## コンテキストに残さない側道質問: `/btw`

> 公式: 「For quick questions that don't need to stay in context, use `/btw`. The answer appears in a dismissible overlay and never enters conversation history」

メイン作業と無関係な疑問（「この関数何のためにあるんだっけ？」「Node のバージョンは？」等）は `/btw` で投げる。回答はオーバーレイ表示され**会話履歴に残らない**ため、本タスクのコンテキストを汚さない。

使い分け:
- 本タスクに関係する質問 → 通常質問（履歴に残す）
- 寄り道質問 / 知識確認 → `/btw`（履歴に残さない）

## チェックポイントと巻き戻し戦略

> 公式: 「Every action Claude makes creates a checkpoint... Instead of carefully planning every move, you can tell Claude to try something risky.」

Claude の各操作は自動でチェックポイントを作成する。これを活用して **「リスクのあるアプローチをまず試し、駄目なら巻き戻す」** スタイルが有効:

- **試行的実装:** 「とりあえずこの方針で書いてみて、駄目なら別案に切り替える」と Claude に伝える
- **失敗時:** `Esc+Esc` → `/rewind` でメッセージ・コード・両方のいずれかを巻き戻す
- **永続性:** チェックポイントはセッションを跨いで保持される（ターミナル閉じても再開可能）

**注意:** チェックポイントは Claude の変更のみを追跡。外部プロセス（手動編集 / git operation）の変更は追跡されない → git の代替にはならない。

## セッション分離

1つのセッションで調査と実装を混在させない。

- **調査セッション**: コードベースの探索、設計検討、技術調査を行い、結果をステアリングドキュメントに記録する
- **実装セッション**: 調査結果（ステアリングドキュメント）を読み込み、実装に集中する

`/fork` を活用して、調査と実装を別セッションに分離することを推奨する。

## 避けるべきアンチパターン

### キッチンシンクセッション

1つのセッションに無関係な複数のタスクを詰め込むこと。タスクごとにセッションを分ける。

### 繰り返し修正ループ

同じエラーに対して同じアプローチで繰り返し修正を試みること。3回同じ修正が失敗したら、アプローチを根本的に見直す。

### 無限探索

明確なゴールなくコードベースを探索し続けること。探索前に「何を知りたいか」を明確にし、得られたら探索を終了する。
