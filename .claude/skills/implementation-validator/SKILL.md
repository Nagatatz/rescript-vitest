# 実装コード検証

実装コードの品質を5つの評価基準で検証し、構造化されたレポートを出力する。

## 使い方

- `/implementation-validator` — 最近変更されたファイルを自動検出して検証
- `/implementation-validator src/path/to/File.ext` — 指定ファイルのみ
- `/implementation-validator src/` — ディレクトリ指定

## 検証手順

### ステップ1: 対象ファイルの特定

引数が指定されている場合はそのファイル（またはディレクトリ配下のソースファイル）を対象とする。

引数がない場合は以下のコマンドで最近変更されたファイルを自動検出する:

```bash
git diff --name-only HEAD~3
```

検出されたファイルをソースコードとその他（設定、ドキュメント等）に分類する。

### ステップ2: スペック・規約の読み込み

以下のドキュメントを Read で読み込み、検証の基準とする。ファイルが存在しない場合はスキップする。

1. **CLAUDE.md** — コーディング規約、コメント規約、アーキテクチャ概要
2. **docs/architecture.md** — アーキテクチャ構成、モジュール責務
3. **docs/functional-design.md** — 機能設計、API仕様
4. **docs/product-requirements.md** — プロダクト要件
5. **.steering/**/requirements.md** — 作業単位の要件（Glob で検索、最新のもの）
6. **.steering/**/design.md** — 作業単位の設計（Glob で検索、最新のもの）

### ステップ3: ビルド・テスト実行

CLAUDE.md に記載されたビルドコマンド・テストコマンドを実行する。
記載がない場合は、プロジェクトのビルドツールを自動検出して実行する。

結果（成功/失敗、エラー内容）を記録する。ビルド失敗の場合も検証を続行し、レポートに含める。

### ステップ4: カバレッジギャップ分析

対象ファイルの検証に加え、プロジェクト全体のテストカバレッジギャップを分析する。

**手順:**

1. ソースディレクトリ配下の全モジュールを Glob で列挙する
2. テストディレクトリ配下のテストファイルを Read し、テストされている関数・モジュールを特定する
3. ソースモジュールとテストの対応を突合し、未テストモジュールを特定する
4. 未テストモジュールに優先度を付与する:

| 優先度 | 基準 | 例 |
|--------|------|-----|
| **高** | ビジネスロジック（計算・変換）で未テスト | 計算処理、データ変換 |
| **中** | データアクセス層で主要 CRUD が未テスト | DB操作、API呼び出し |
| **低** | インフラ層・薄いラッパー | ログ、型定義のみ |

**レポートに含める内容:**

```
### カバレッジギャップ

| モジュール | テスト状態 | 優先度 | 備考 |
|-----------|-----------|--------|------|
| calculator.ext | ✅ 6テスト | - | 基本パス網羅 |
| parser.ext | ✅ 8テスト | - | ユーティリティ関数網羅 |
| allocator.ext | ❌ 未テスト | 高 | 213行の計算ロジック |
| repository.ext | ❌ 未テスト | 中 | CRUD操作 |
| downloader.ext | ❌ 未テスト | 低 | 外部連携（統合テスト向き） |
```

### ステップ5: 5つの評価基準で検証

対象ファイルをすべて Read で読み込み、以下の5基準で評価する。各基準は1〜5のスコアで採点する。

### ステップ6: レポート生成

後述のレポートフォーマットに従い、検証結果を出力する。カバレッジギャップ分析の結果もレポートに含める。

### ステップ7: 改善の優先順位を提示

指摘事項を影響度順に並べ、どこから着手すべきかを明示する。

## 5つの評価基準

### 1. スペック準拠（Spec Compliance）

設計ドキュメント・規約に沿った実装になっているか。

**チェック項目:**
- PRD要件（docs/product-requirements.md）が正しく実装されているか
- データモデルが機能設計（docs/functional-design.md）と一致しているか
- アーキテクチャ構造を遵守しているか（レイヤー間の依存関係が正しいか）
- APIの仕様が設計と一致しているか
- .steering/ の要件・設計に沿っているか

**注意**: docs/ が未整備の場合は CLAUDE.md のみで評価し「(一部ドキュメント未整備)」と付記する。

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | 全要件を正確に実装。アーキテクチャ構造も完全に遵守 |
| 4 | ほぼ準拠。軽微な差異のみ |
| 3 | 主要要件は実装済みだが、一部に設計との乖離あり |
| 2 | 重要な要件の未実装、またはレイヤー違反が複数 |
| 1 | 設計と大幅に乖離。要件の大半が未実装 |

### 2. コード品質（Code Quality）

言語のベストプラクティスとプロジェクト規約に沿っているか。

**チェック項目:**
- プロジェクトの言語・フレームワーク固有のベストプラクティスに従っているか
- 命名規則が統一されているか
- 単一責任の原則を守っているか。重複コードがないか
- コメント規約（CLAUDE.md 参照）に従っているか
- エラーハンドリングが適切か

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | 全規約に準拠。ベストプラクティスが模範的 |
| 4 | ほぼ準拠。軽微なスタイル違反のみ |
| 3 | 概ね良好だが、規約違反が散見される |
| 2 | ベストプラクティスに反する実装が多い |
| 1 | 規約をほぼ無視した実装 |

### 3. テストカバレッジ（Test Coverage）

新規・変更コードに対するテストが十分か。

**チェック項目:**
- テストが全件パスするか
- 新規・変更されたコードに対応するテストが存在するか（Grep でテストディレクトリ配下を検索）
- エッジケースがテストされているか（空リスト、null/None、0件、境界値）
- エラーケースのテストがあるか
- テストの命名が内容を適切に表しているか

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | 全ロジックにテストあり。エッジケース・エラーケースも網羅 |
| 4 | 主要ロジックにテストあり。一部のエッジケースが未テスト |
| 3 | 基本パスのテストはあるが、エッジケースが不足 |
| 2 | テストが少なく、主要ロジックもカバーされていない |
| 1 | テストが存在しない、またはテストが全件失敗 |

### 4. セキュリティ（Security）

セキュリティ上の脆弱性がないか。

**チェック項目:**
- **入力バリデーション**: 外部入力（APIパラメータ、ファイルアップロード、ユーザー入力）が適切に検証されているか
- **インジェクション対策**: SQL、コマンド、XSS等のインジェクションが防止されているか
- **ハードコード秘密情報**: `password` / `secret` / `api_key` / `token` / `credential` を Grep で検索
- **パストラバーサル**: ファイルパスの組み立てに外部入力が使われていないか
- **依存関係**: 既知の脆弱性を含むライブラリを使用していないか

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | セキュリティ問題なし。入力検証・インジェクション対策も適切 |
| 4 | 概ね安全。軽微なリスクのみ |
| 3 | 一部にリスクあり（バリデーション漏れ等）だが、重大な脆弱性はない |
| 2 | インジェクションリスクまたは秘密情報の露出あり |
| 1 | 複数の重大な脆弱性が存在 |

### 5. パフォーマンス（Performance）

明らかなパフォーマンス問題がないか。

**チェック項目:**
- **N+1問題**: ループ内でDB接続・クエリ・API呼び出しを実行していないか
- **リソース解放**: 接続やストリーム等のリソースが適切に閉じられているか
- **不要な再計算**: 計算結果のキャッシュが必要な箇所で毎回再計算していないか
- **アルゴリズム効率**: 明らかに非効率なアルゴリズム（O(n²)で済むところをO(n³)等）がないか
- **メモリ効率**: 不要に大きなデータをメモリに保持していないか

**スコア基準:**
| スコア | 基準 |
|--------|------|
| 5 | パフォーマンス問題なし。リソース管理も適切 |
| 4 | 概ね良好。軽微な最適化の余地のみ |
| 3 | 一部に非効率な処理あり（影響は限定的） |
| 2 | N+1問題やリソースリークのリスクあり |
| 1 | 重大なパフォーマンス問題が複数存在 |

## レポートフォーマット

以下の形式で検証結果を出力する。

### 検証対象

対象ファイルと参照ドキュメントを一覧にする。

```
**対象ファイル:**
- src/path/to/Calculator.ext
- src/path/to/Parser.ext

**参照ドキュメント:**
- ✅ CLAUDE.md
- ✅ docs/architecture.md
- ⚠️ docs/functional-design.md （未整備）
- ✅ .steering/20260215-xxx/requirements.md
```

### ビルド・テスト結果

各コマンドの実行結果を記載する。

```
| コマンド | 結果 | 詳細 |
|---------|------|------|
| ビルド | ✅ パス | 警告なし |
| テスト | ✅ パス | 15 tests passed |
```

### 評価サマリー

全ファイルの総合スコアを一覧にする。

```
| 基準 | スコア | 概要 |
|------|--------|------|
| スペック準拠 | 4 | 主要要件を実装済み |
| コード品質 | 5 | ベストプラクティスに準拠、模範的 |
| テストカバレッジ | 3 | 基本パスのテストあり。エッジケースが不足 |
| セキュリティ | 5 | インジェクション対策・入力検証も適切 |
| パフォーマンス | 4 | リソース管理適切。軽微な最適化余地あり |
| **総合** | **4.2** | |
```

総合スコアは5基準の平均値（小数点第1位まで）。

### 良い点

模範的な実装を具体的にリストアップする。

### 指摘事項

改善が必要な箇所を [必須] / [推奨] に分類して列挙する。

```
#### [必須] 指摘事項のタイトル

- **対象**: ファイル名:行番号
- **基準**: 該当する評価基準名
- **理由**: なぜ問題なのかの説明
- **改善案**: 具体的な改善方法
```

[必須] と [推奨] の分類基準:
- **[必須]**: ビルド失敗、テスト失敗、セキュリティ脆弱性、重大な設計違反
- **[推奨]**: コード品質の改善、テスト追加、パフォーマンス最適化

### 改善の優先順位

指摘事項を影響度の高い順に並べ、着手順序を提示する。

```
1. **最優先**: ビルド・テスト失敗の修正
2. **高**: セキュリティ脆弱性の修正
3. **中**: テストカバレッジの拡充
4. **低**: コード品質改善
```
